# Lab11 为 Sports 应用添加大屏自适应布局实验报告

## 一、实验目的

1. 理解 `WindowSizeClass` 窗口尺寸分类机制，掌握不同设备屏幕尺寸的适配思路。
2. 学会使用 `WindowWidthSizeClass` 判断设备宽度类型，实现界面布局的动态切换。
3. 掌握 Compose 中“列表-详情（List-Detail）”双栏布局的实现方式，提升大屏设备的显示效果。
4. 完成顶部导航栏在不同屏幕模式下的差异化适配，实现标题与返回按钮的动态显示。
5. 根据不同布局结构设计系统返回键逻辑，优化应用交互体验。
6. 巩固 Jetpack Compose 状态管理、ViewModel 数据共享以及 `Row + weight` 自适应布局等核心知识。

------

## 二、实验原理

### 2.1 WindowSizeClass 窗口尺寸分类

`material3-window-size-class` 是 Google 官方提供的自适应布局工具库，可自动根据窗口大小对设备进行分类，无需手动获取屏幕分辨率，从而实现跨设备界面适配。

其中 `WindowWidthSizeClass` 按窗口宽度分为以下三种类型：

1. **Compact（紧凑型）**
   主要适用于手机竖屏设备，横向空间有限，通常采用单页面跳转布局。
2. **Medium（中等型）**
   适用于折叠屏设备或小尺寸横屏设备，本实验中仍采用单窗格布局。
3. **Expanded（展开型）**
   适用于平板及大屏设备，横向空间充足，可实现列表与详情同屏展示。

### 2.2 SportsContentType 枚举类

项目通过自定义枚举类统一管理界面展示模式，简化自适应布局判断逻辑：

- `ListOnly`：单列表模式，仅用于手机端，通过页面跳转查看详情。
- `ListAndDetail`：列表与详情双栏模式，用于大屏设备，实现左右同屏展示。

### 2.3 List-Detail 双栏布局原理

实验遵循 Material3 自适应设计规范，使用 `Row` 横向布局配合 `Modifier.weight()` 按比例划分界面区域：

- 左侧显示运动列表，作为导航区域；
- 右侧展示运动详情内容；

该布局能够充分利用大屏横向空间，提高内容浏览效率与用户体验。

## 三、实验内容与实验步骤

### 3.1 实验内容

基于已有 Sports 运动资讯应用，新增大屏自适应功能，实现：

- 手机端单页面跳转模式；
- 平板端列表与详情双栏同屏模式；
- 导航栏样式适配；
- 返回键逻辑适配；

最终实现应用在不同设备上的自适应运行。

### 3.2 实验步骤

1. **导入项目**
   在 Android Studio 中打开 Sports 起始项目，完成 Gradle 同步后运行程序，熟悉原有列表与详情页面逻辑。
2. **计算窗口尺寸**
   修改 `MainActivity.kt`，引入窗口尺寸相关 API，通过 `calculateWindowSizeClass()` 获取当前窗口宽度类型，并将结果传递给顶层 `SportsApp` 组合函数。
3. **实现布局模式判断**
   在 `SportsApp` 中根据窗口宽度类型进行判断：
   - Expanded 模式启用双栏布局；
   - 其他模式保持原有单窗格布局。
4. **编写双栏布局组件**
   创建 `SportsListAndDetails` 组合函数，使用 `Row` 实现左右分栏布局，并通过 `weight` 设置页面占比，实现详情内容实时刷新。
5. **适配顶部导航栏**
   修改 `SportsAppBar`：
   - 大屏模式固定显示应用名称并隐藏返回按钮；
   - 小屏模式保留标题切换与返回按钮逻辑。
6. **适配返回键逻辑**
   - 小屏模式下，返回键用于页面回退；
   - 大屏模式下，由于列表与详情同屏展示，返回键直接退出应用。
7. **运行与测试**
   使用手机模拟器与 Pixel Tablet 平板模拟器分别测试布局切换、导航栏样式、详情刷新及返回键逻辑，并完成实验截图保存。

## 四、核心代码设计说明

### 4.1 窗口尺寸传递流程

应用在 `MainActivity` 中统一获取窗口尺寸类型，并作为参数传递至顶层组合函数，实现全局统一的布局适配标准。

### 4.2 双栏布局比例设计

双栏布局通过 `Modifier.weight()` 设置页面比例：

- 左侧运动列表：`1f`，约占整体宽度的三分之一；
- 右侧运动详情：`2f`，约占整体宽度的三分之二。

这样既能保证导航区域简洁清晰，又能为详情内容提供更大的展示空间，符合大屏设备的使用习惯。

### 4.3 顶部导航栏差异化设计

1. **ListAndDetail（大屏模式）**
   固定显示应用名称，隐藏返回按钮。
2. **ListOnly（小屏模式）**
   列表页显示应用名称，详情页显示对应标题，并保留返回按钮。

### 4.4 返回键交互策略

1. **手机端**
   详情页点击返回键后回到列表页，符合移动端常规操作习惯。
2. **平板端**
   列表与详情始终同屏展示，无页面层级关系，因此按下系统返回键时直接退出应用。

## 六、实验中遇到的问题及解决方案

### 1. 实验性 API 报错

**问题：** 使用窗口尺寸相关 API 时出现实验性接口报错。

**解决方案：**
在 Activity 与相关组合函数上添加：

```kotlin
@OptIn(ExperimentalMaterial3WindowSizeClassApi::class)
```

解除实验性 API 限制。

### 2. 大屏模式下详情页无法刷新

**问题：** 点击左侧列表后，右侧详情内容未同步更新。

**解决方案：**
使用 ViewModel 统一管理选中状态，列表点击后仅修改全局状态数据，由详情页自动监听并刷新。

### 3. 大屏模式仍显示返回按钮

**问题：** 双栏模式下导航栏依旧显示返回按钮。

**解决方案：**
在导航栏中新增布局模式判断，当处于双栏模式时隐藏返回图标。

### 4. 大屏返回键逻辑异常

**问题：** 大屏模式下按返回键仍会返回列表页面。

**解决方案：**
在双栏布局中单独监听系统返回事件，覆盖原有页面回退逻辑，实现直接退出应用。

### 5. 双栏布局显示错乱

**问题：** 页面在部分大屏设备中出现内容挤压。

**解决方案：**
取消固定宽高设置，统一采用 `weight` 按比例分配空间，实现自适应布局。

## 七、实验总结

本次实验成功为 Sports 应用实现了大屏自适应布局功能，掌握了 `WindowSizeClass` 尺寸分类、Compose 双栏布局、导航栏适配以及系统返回键逻辑处理等核心开发技术。

通过针对手机与平板设备采用不同布局方案，既保留了移动端简洁直观的页面跳转体验，又充分发挥了大屏设备的显示优势，实现了列表与详情内容的同屏展示，有效提升了应用的多设备兼容性与用户体验。

同时，本实验进一步巩固了 Jetpack Compose 状态管理、ViewModel 数据共享、自适应布局设计等知识，为后续进行 Android 多终端适配开发奠定了良好基础。
