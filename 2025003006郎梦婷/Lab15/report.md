# Lab15 Flight Search 应用实验报告
## 一、实验基本信息
1. **实验名称**：Flight Search 航班搜索应用开发
2. **实验目的**
    - 掌握 Android Room 框架实现关系型数据库读写、模糊查询、联表查询。
    - 学会使用 Preferences DataStore 实现轻量数据持久化与状态保存。
    - 基于 Jetpack Compose 搭建复杂界面，完成自动补全、列表展示、界面状态切换。
    - 结合 ViewModel + Flow 管理 UI 状态，实现数据异步加载与界面响应。
3. **开发环境**：Android Studio、Kotlin、Jetpack Compose、Room、DataStore
4. **实验功能**：机场搜索自动补全、航班列表展示、航班收藏/取消、搜索文本持久化、空搜索展示收藏列表。

## 二、Entity 实体类设计说明
本次实验对应数据库两张表：`airport`（机场表）、`favorite`（收藏航班表），使用 Room Entity 完成表映射。

### 1. Airport 实体类（对应 airport 表）
#### 表结构映射
| 数据库字段 | 实体属性 | 类型 | 约束 |
| ---- | ---- | ---- | ---- |
| id | id | Int | 主键、自增 |
| iata_code | iataCode | String | 机场三字码 |
| name | name | String | 机场名称 |
| passengers | passengers | Int | 年客流量 |

#### 设计要点
- 使用 `@Entity(tableName = "airport")` 绑定数据库表名。
- `id` 标注为主键，设置 `autoGenerate = true`。
- 字段名通过 `@ColumnInfo` 与数据库原生字段对齐，避免命名冲突。
- 提供数据解析、实例构造方法，供 DAO、ViewModel、UI 调用。

### 2. Favorite 实体类（对应 favorite 表）
#### 表结构映射
| 数据库字段 | 实体属性 | 类型 | 约束 |
| ---- | ---- | ---- | ---- |
| id | id | Int | 主键、自增 |
| departure_code | departureCode | String | 出发地 IATA 码 |
| destination_code | destinationCode | String | 目的地 IATA 码 |

#### 设计要点
- 仅存储**机场三字码**，不冗余存储机场名称，遵循数据库范式。
- 航班名称需通过**联表查询**从 `airport` 表关联获取。
- 作为收藏记录表，用于标记用户收藏的航线。

### 3. 拓展实体（联表查询结果）
由于收藏列表需要同时展示**出发/目的地代码+名称**，额外创建数据类封装联表查询结果，用于 UI 展示。

## 三、DAO 数据访问层设计说明
基于 Room DAO 接口，定义所有数据库操作方法，结合 SQL 语句实现**模糊查询、普通查询、联表查询、增删改**。

### 1. 自动补全查询（机场模糊搜索）
#### 功能需求
根据用户输入内容，**模糊匹配 iata_code 和 name**，按客流量 `passengers` 降序排序，优先展示热门机场。
#### SQL 语句
```sql
SELECT * FROM airport
WHERE iata_code LIKE :query OR name LIKE :query
ORDER BY passengers DESC
```
#### 实现说明
1. 使用 SQL `LIKE` 关键字实现模糊匹配，参数拼接 `%`（`%query%`）实现**全匹配搜索**。
2. 同时检索机场代码和机场名称，满足两种输入习惯。
3. 按客流量降序排序，优化自动补全展示优先级。
4. 返回 `Flow<List<Airport>>`，实现数据实时监听，输入变化自动刷新列表。

### 2. 航班列表查询
#### 功能需求
根据选中的出发机场 IATA 码，查询**所有其他机场**作为目的地（排除自身）。
#### 实现说明
1. 查询 `airport` 表，过滤条件：`iata_code != 出发地编码`。
2. 逻辑规则：默认所有机场之间互通航班，仅排除同机场航线。
3. 同样返回 Flow 数据流，选中机场后自动加载航班列表。

### 3. 收藏航线增删查
1. **查询所有收藏**：使用**内连接（INNER JOIN）** 联表查询 `favorite` 与 `airport` 表，将代码转为机场名称，供收藏列表展示。
2. **添加收藏**：向 `favorite` 表插入出发地、目的地 IATA 码。
3. **删除收藏**：根据出发+目的地编码匹配，删除对应收藏记录。
4. 所有收藏相关操作封装为挂起函数（suspend），在协程中执行，避免主线程阻塞。

### 4. 联表查询设计（核心）
#### 使用场景
`favorite` 表仅存储机场编码，UI 需要展示机场名称，因此必须联表查询。
#### SQL 逻辑
将 `favorite` 表分别与 `airport` 表做两次内连接：
- 别名 `departure`：关联出发地机场信息
- 别名 `destination`：关联目的地机场信息
#### 作用
- 减少数据库冗余，一张收藏表只存核心关联字段。
- 动态关联基础数据表，保证数据一致性。
- 一次性查询出页面所需全部字段，减少多次数据库访问，提升性能。

## 四、SQL LIKE 关键字使用说明
### 1. 作用
`LIKE` 是 SQL 模糊查询关键字，用于**匹配部分字符串**，适用于搜索、自动补全场景，区别于精准等值查询。

### 2. 通配符规则
- `%`：匹配**任意长度任意字符**（包括空字符）。
  - `%abc`：匹配以 `abc` 结尾的内容。
  - `abc%`：匹配以 `abc` 开头的内容。
  - `%abc%`：匹配**包含 abc** 的所有内容（本实验使用该规则）。
### 3. 本实验使用方式
1. 前端获取用户输入文本 `searchText`。
2. 代码拼接通配符：`"%$searchText%"`。
3. 将拼接后的字符串作为参数传入 Room SQL 语句。
4. 同时匹配 `iata_code` 和 `name` 两个字段，实现全维度搜索。

### 4. 优势
用户无需输入完整机场名/编码，输入片段即可得到联想结果，完美实现**输入自动补全**功能。

## 五、联合查询（JOIN）实现与作用
### 1. 实现方式
采用 **INNER JOIN（内连接）**，多表关联查询：
1. 主表：`favorite`（收藏表，存储航线编码）。
2. 关联表1：`airport AS departure`（获取出发机场详情）。
3. 关联表2：`airport AS destination`（获取目的机场详情）。
4. 关联条件：`favorite.departure_code = departure.iata_code`、`favorite.destination_code = destination.iata_code`。

### 2. 核心作用
1. **解耦数据表**：基础机场信息统一存放在 `airport` 表，收藏表仅做关联记录，符合数据库设计规范。
2. **数据整合**：单次查询同时获取「收藏关系 + 机场名称/编码」，减少多次数据库请求。
3. **数据实时同步**：若机场信息更新，联表查询可自动获取最新数据，无需修改收藏表。

### 3. 适用场景
本实验**搜索框为空时**，加载收藏航班列表，全程依赖联表查询渲染完整航线信息。

## 六、Preferences DataStore 实现与使用场景
### 1. 技术定位
Preferences DataStore 是 Jetpack 推荐的**轻量级键值对持久化方案**，替代传统 SharedPreferences，基于 Flow 实现异步、线程安全的数据读写，适合存储简单配置、临时状态。

### 2. 本实验使用场景
专门持久化**用户搜索框文本**，实现两大需求：
1. 应用重启后，自动回填上次的搜索内容。
2. 若上次退出时搜索框为空，重启后直接展示收藏列表。

### 3. 具体实现步骤
1. **定义偏好键**：创建字符串类型 Key，用于存储搜索文本。
2. **初始化 DataStore 实例**：全局单例，绑定应用上下文。
3. **读取数据**：通过 `dataStore.data` 获取 Flow 流，映射为搜索文本，ViewModel 收集状态。
4. **保存数据**：提供挂起函数，在协程中调用 `dataStore.edit {}` 更新文本。
5. **联动逻辑**：用户输入变化时实时保存，应用启动时优先读取本地缓存回填输入框。

### 4. 优势
- 异步操作，不阻塞主线程；
- 基于 Flow 可监听数据变化；
- 相比 SP，API 更规范、兼容性更强。

## 七、ViewModel 状态管理设计
### 1. 核心职责
作为**数据层与 UI 层的中间层**，统一管理所有 UI 状态、数据流、业务逻辑，保证配置变更（屏幕旋转）、页面重建后状态不丢失。

### 2. 状态分类与管理
1. **搜索文本状态**
    - 用 `MutableStateFlow` 维护输入框文本。
    - 文本变化时：触发自动补全查询、调用 DataStore 持久化。
2. **自动补全列表状态**
    - 监听搜索文本，文本非空时触发机场模糊查询，更新联想列表。
3. **航班列表状态**
    - 用户选中补全项后，查询对应出发地的所有航班，更新航班列表。
4. **收藏列表状态**
    - 搜索框**为空**时，自动加载联表查询后的收藏航线。
5. **收藏操作逻辑**
    - 封装添加/删除收藏方法，调用 DAO 协程函数，操作完成后刷新列表。

### 3. 数据流架构
`数据库/DataStore(Flow)` → `ViewModel(collect + StateFlow)` → `Compose UI(collectAsState)`
- 所有数据异步加载，UI 仅负责展示，不直接操作数据库与持久化组件。
- ViewModel 生命周期独立于 Activity，屏幕旋转等场景**状态自动保留**。

### 4. 协程使用
所有耗时操作（数据库读写、DataStore 读写）均在 ViewModel 内置协程作用域 `viewModelScope` 中执行，页面销毁时自动取消协程，避免内存泄漏。

## 八、Jetpack Compose UI 切换逻辑说明
整体页面基于 `LazyColumn`、输入框、按钮组合实现，根据**搜索文本内容**自动切换不同视图，核心分为 4 种界面状态：

### 1. 初始状态（搜索框为空）
- 隐藏自动补全、航班列表。
- 加载并展示**收藏航班列表**。
- 逻辑：空输入 = 展示收藏。

### 2. 输入中状态（搜索框有文本，未选中机场）
- 监听输入文本，触发数据库模糊查询。
- 在输入框下方弹出**自动补全机场列表**。
- 使用布局层级（Box）实现浮层效果，列表随输入实时刷新。

### 3. 选中机场状态
- 点击补全列表项后，记录选中机场。
- 隐藏自动补全列表。
- 展示**该机场出发的所有航班列表**。
- 每个航班条目展示：出发地 IATA 码、出发机场名、目的地 IATA 码、目的地名 + 收藏按钮。

### 4. 收藏按钮交互
- 动态判断当前航线是否已收藏，切换按钮文字/图标。
- 点击按钮：执行「添加收藏」或「删除收藏」，列表实时刷新。

### 5. 布局技术点
1. 使用 `TextField` 实现搜索输入框。
2. `LazyColumn` 高效渲染长列表（自动补全、航班、收藏列表）。
3. 利用 Compose 状态驱动 UI，数据变化自动刷新界面，无需手动更新视图。
4. 全局监听 DataStore 缓存，应用启动自动回填搜索文本，恢复上次界面。

## 九、实验问题与解决过程
### 问题1：预置数据库无法加载（createFromAsset 失效）
- **现象**：应用启动后查询不到任何机场数据，数据库为空。
- **原因**：assets 目录层级错误，数据库文件存放路径不匹配代码配置。
- **解决**：严格按照要求创建 `assets/database/` 目录，将 `flight_search.db` 放入该路径；Room 数据库初始化时正确指定 `createFromAsset("database/flight_search.db")`。

### 问题2：LIKE 模糊查询不生效
- **现象**：输入内容后无任何联想结果。
- **原因**：未拼接 `%` 通配符，直接使用原始文本查询，变成精准匹配。
- **解决**：代码中拼接 `"%$inputText%"` 再传入 SQL 参数，实现模糊匹配。

### 问题3：航班列表出现“出发地=目的地”的无效航线
- **现象**：列表中出现机场飞往自身的条目。
- **原因**：航班查询未做过滤判断。
- **解决**：在 SQL 中增加过滤条件 `iata_code != :departureCode`，排除同机场航线。

### 问题4：DataStore 读取不到历史搜索文本
- **现象**：重启应用后输入框无法自动回填内容。
- **原因**：ViewModel 未在启动时主动收集 DataStore 数据流，UI 未初始化文本状态。
- **解决**：ViewModel 初始化时收集 `searchTextFlow`，将本地缓存文本赋值给输入框状态。

### 问题5：收藏列表只显示编码，无机场名称
- **现象**：收藏条目只有 `departure_code`、`destination_code`，没有中文/英文机场名。
- **原因**：未使用联表查询，仅查询了 `favorite` 单表。
- **解决**：改写 DAO 查询语句，使用 `INNER JOIN` 关联两张表，封装联表结果数据类供 UI 使用。

### 问题6：主线程访问数据库崩溃
- **现象**：点击按钮、输入内容时应用闪退，提示主线程不能访问数据库。
- **原因**：Room 数据库操作未放在协程中执行。
- **解决**：将增删改查方法定义为 `suspend` 挂起函数，在 `viewModelScope.launch` 协程中调用。

### 问题7：屏幕旋转后页面状态丢失
- **现象**：旋转屏幕后，搜索文本、列表全部重置。
- **原因**：状态直接定义在 Composable 中，Activity 重建后状态销毁。
- **解决**：所有 UI 状态统一交由 ViewModel 管理，ViewModel 不受配置变更影响。
