# 掷骰子应用开发实验报告

## 一、应用界面结构说明
本应用基于 **Jetpack Compose** 实现，是一个单页面交互式掷骰子程序。界面结构清晰、简洁，主要包含以下部分：

### 1. 顶层容器
使用 `Scaffold` 作为页面根容器，适配系统状态栏与导航栏，保证布局完整显示。

### 2. 核心布局
使用 `Column` 垂直布局，将所有子元素垂直排列，并设置为**屏幕水平+垂直居中**，适配不同尺寸的手机屏幕。

### 3. 界面元素（从上到下）
- **骰子图片（Image）**：显示当前点数对应的骰子图案，是界面的核心视觉元素；
- **间距（Spacer）**：设置 16dp 的空白间隔，优化图片与按钮之间的视觉效果；
- **按钮（Button）**：用户点击后随机生成骰子点数，触发界面图片更新。

### 4. 组件结构
- `MainActivity`：应用入口，负责加载主题与主界面；
- `DiceRollerApp`：主界面根组件，统一管理页面；
- `DiceWithButtonAndImage`：封装所有 UI 展示与交互逻辑。

## 二、使用 Compose 状态保存骰子结果
在 Jetpack Compose 中，**界面由状态驱动**，状态变化会自动触发界面刷新。本应用使用可观察状态保存骰子点数：

### 1. 状态声明代码
```kotlin
var result by remember { mutableIntStateOf(1) }
```

### 2. 关键说明
- `mutableIntStateOf(1)`：创建可观察的整型状态，初始值为 1（默认显示 1 点骰子），可被 Compose 自动监听变化；
- `remember`：保证屏幕旋转、界面重组时，状态值不丢失，持续保存当前骰子点数；
- `var` 修饰：因需要在按钮点击时修改点数，必须使用可变变量，`val` 只读变量无法更新。

### 3. 核心作用
该状态 `result` 是应用核心数据，存储当前骰子点数，所有界面刷新均围绕其变化执行。

## 三、根据点数切换图片资源
通过 `when` 条件表达式，将骰子点数与本地图片资源一一绑定，实现图片自动切换。

### 1.资源匹配代码
```
kotlin
val imageResource = when (result) {
    1 -> R.drawable.dice_1
    2 -> R.drawable.dice_2
    3 -> R.drawable.dice_3
    4 -> R.drawable.dice_4
    5 -> R.drawable.dice_5
    else -> R.drawable.dice_6
}
```

### 2. 执行逻辑
- 当 `result` 点数发生变化时，Compose 自动触发界面重组；
- `when` 表达式根据最新点数，匹配对应的图片资源 ID；
- `Image` 组件通过 `painterResource(imageResource)` 加载对应图片，完成界面更新；
- `else` 分支覆盖所有异常情况，确保程序健壮，避免资源找不到的错误。

## 四、断点设置与观察内容
本次实验在核心逻辑处设置 3 个断点，用于调试程序执行流程与变量变化。

### 1. 断点 1：状态初始化行
- 位置：`var result by remember { mutableIntStateOf(1) }`
- 观察目标：验证程序启动时，状态是否正常创建，初始点数是否为 1。

### 2. 断点 2：按钮点击事件内部
- 位置：`result = (1..6).random()`
- 观察目标：
  - 点击按钮时，程序是否正常触发点击逻辑；
  - 随机生成的点数是否在 1~6 的合法范围内；
  - `result` 状态是否被成功赋值更新。

### 3. 断点 3：when 表达式图片匹配行
- 位置：`val imageResource = when (result)`
- 观察目标：
  - 当前 `result` 点数是否正确；
  - 程序是否匹配到与点数对应的图片资源 ID。

## 五、Step Into、Step Over、Step Out 的使用体会
在调试过程中，结合 Android Studio 三大调试功能，高效排查问题，具体体会如下：

### 1. Step Over（单步跳过）
- 使用场景：逐行执行当前函数内的代码，不进入子函数内部；
- 体会：本次实验最常用的调试方式，适合观察按钮点击、状态赋值、图片匹配等线性业务逻辑，能清晰看到每一步变量的变化，快速定位基础问题。

### 2. Step Into（单步跳入）
- 使用场景：进入当前行调用的函数或系统源码内部；
- 体会：可深入理解 Compose 状态更新、随机数生成的底层逻辑，适合排查深层次 bug，但需避免随意跳入系统源码，防止调试流程混乱。

### 3. Step Out（单步跳出）
- 使用场景：快速退出当前子函数，返回到调用处；
- 体会：当不小心跳入系统源码时，可快速回到自身业务代码，大幅提升调试效率，避免在无关代码中浪费时间。

### 4. 整体总结
三者配合使用，日常调试优先用 Step Over，需深究底层逻辑用 Step Into，需快速返回上层用 Step Out，可高效完成所有调试工作。

## 六、遇到的问题与解决过程
本次开发中遇到多个问题，均通过排查分析完成解决，具体如下：

### 1. 问题 1：大量语法报错
- 现象：出现 Function must have a body、Expecting a top level declaration 等错误；
- 原因：代码大括号不匹配，状态、布局代码写在函数外部，不符合语法规范；
- 解决：规范代码结构，将所有 UI 和逻辑代码包裹在 DiceWithButtonAndImage 函数体内，补全所有缺失的括号。

### 2. 问题 2：按钮点击后图片不刷新
- 现象：点击按钮后，点数未更新，图片无变化；
- 原因：使用 val result 声明只读状态，无法修改点数；
- 解决：将 val 改为 var，并使用 mutableIntStateOf 创建可观察状态，支持点数更新。

### 3. 问题 3：拼写错误导致编译失败
- 现象：mutableStateof、spacer、fillMaxsize 等关键词报红；
- 原因：大小写错误、组件名称拼写错误；
- 解决：修正为正确拼写，即 mutableStateOf、Spacer、fillMaxSize。

### 4. 问题 4：随机数语法错误
- 现象：1≤ .. ≤6 写法不合法，编译报错；
- 解决：改为 Kotlin 标准写法 (1..6).random()，实现 1~6 范围内的随机数生成。

### 5. 问题 5：Preview 注解无法识别
- 现象：@Preview 报红，提示 Unresolved reference 'Preview'；
- 原因：缺少 @Preview 注解的导入语句；
- 解决：添加导入 import androidx.compose.ui.tooling.preview.Preview。

## 七、实验结论

### 1. 按钮点击后图片自动刷新的原因
`result` 是 Compose 可观察状态，当按钮点击修改其值时，Compose 会自动触发界面重组，重新执行 `when` 表达式匹配图片资源，`Image` 组件加载新图片，实现无手动刷新的自动更新，这是 Compose「状态驱动 UI」核心设计思想的体现。

### 2. 调试器中变量值与界面结果的一致性
调试器中观察到的 `result` 变量值、图片资源 ID，与界面显示的骰子点数、图片完全一致，证明状态管理、逻辑判断、UI 渲染均正常工作，程序无异常。

### 3. 实验总结
通过本次开发，我掌握了 Jetpack Compose 状态管理、条件渲染、按钮交互及基础调试技巧，深刻理解了「状态驱动 UI」的核心逻辑，能够独立完成简单的交互式 Android 应用开发，同时熟悉了 Android Studio 调试工具的使用方法，提升了代码排查与问题解决能力。
