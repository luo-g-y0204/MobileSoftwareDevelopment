# 移动端软件开发期末大作业要求

## 一、作业背景

本学期实验已经覆盖 Kotlin 基础语法、Jetpack Compose 界面开发、Material 3、状态管理、Navigation、多屏适配、协程、网络请求、Repository、Room 数据库和 DataStore 等内容。

期末大作业要求大家综合运用本学期所学知识，**独立设计并实现一个完整的 Android 应用**。与平时 Lab 不同，本次大作业不规定统一题目，大家可以结合自己的兴趣、校园生活或实际需求自拟 App 主题。

本次作业重点考查的不是题目是否复杂，而是能否把一个 Android 应用从界面、状态、数据、交互、持久化到文档说明完整地组织起来。

**关键要求：**

- 题目自选，应用类型不限，但必须是完整 Android App。
- 必须满足下方必做技术要求。
- 必须使用网络功能，至少从真实 API 或 Mock API 获取数据，并在应用核心流程中实际使用。
- 功能可以保持简单，但不能只有静态界面或写死数据。
- 必须独立完成，不允许多人合作。
- 本项目允许并鼓励使用 AI Agent 参与完成。请在实验报告中说明是否使用了 AI，并注明使用了什么 AI 工具；此部分只用于了解开发过程，不影响分值。
- 可参考 Lab 的思路和结构，但不能直接复制改名提交。
- 应用必须能在模拟器或真机上正常运行。

## 二、选题方向

可以选择以下方向，也可以自拟题目。

### 推荐方向

**工具类应用**
- 记账本、待办清单、习惯打卡、日记本、课程表、倒计时、番茄钟

**内容管理类应用**
- 图书管理、电影清单、音乐收藏、食谱管理、笔记应用、学习卡片

**校园生活类应用**
- 校园活动助手、课程与作业管理、二手物品管理、失物招领、社团成员管理、自习室预约

**健康运动类应用**
- 运动记录、健康打卡、体重管理、饮水提醒、睡眠记录

**娱乐游戏类应用**
- 猜谜游戏、记忆卡片、答题闯关、成语接龙、简单棋牌

**其他**
- 旅行计划、公交查询、航班记录、天气记录、地点收藏

### 不建议选择的类型

- 纯个人主页或静态展示 App
- 只有一个页面的简单应用
- 简单计算器或只有按钮点击的 Demo
- 直接改名复刻某一次 Lab
- WebView 套壳应用
- 只有界面、没有数据层和状态管理的应用

### 自拟题目要求

- 有明确的用户场景和实用价值
- 至少包含 2 个主要页面，其中至少包含一个列表页；另一个页面可以是详情页、编辑页、设置页或网络数据页
- 有真实数据读写、网络请求、状态管理和用户操作流程
- 网络数据必须服务于应用功能，不能只添加依赖或写一个未使用的请求示例
- 能够在报告中说明应用的使用场景、目标用户和核心功能

## 三、必做技术要求

以下内容是基本验收要求。缺少 Compose、Room、ViewModel、Navigation、DataStore、网络请求或核心数据持久化，会严重影响成绩。

### 1. UI 层

- [ ] 使用 Kotlin + Jetpack Compose 开发全部界面，不允许使用 XML 布局
- [ ] 使用 Material 3 组件库（Card、Button、TextField、TopAppBar、FloatingActionButton、Snackbar、Dialog 等）
- [ ] 自定义 Material 主题，包括颜色、字体或形状中的至少一类
- [ ] 支持浅色模式和深色模式，至少能跟随系统主题正常显示
- [ ] 应用至少包含 2 个主要页面，其中至少包含一个列表页
- [ ] 使用 Compose Navigation 实现页面导航
- [ ] 使用 LazyColumn 或 LazyVerticalGrid 展示数据列表
- [ ] 页面布局间距、对齐、圆角和视觉层级基本符合 Material Design 规范
- [ ] 列表为空、加载中或出错时，有合理的界面反馈

**参考 Lab：** Lab2–8（Compose UI）、Lab10（Navigation）

---

### 2. 数据层

- [ ] 使用 Room 数据库存储核心数据
- [ ] Room 数据库至少包含 2 个 Entity / 表，字段设计合理
- [ ] 正确使用 `@Entity`、`@PrimaryKey`、`@ColumnInfo` 等注解
- [ ] 创建 DAO 接口，定义查询、插入、更新、删除等方法
- [ ] 核心 Entity 实现完整 CRUD 操作
- [ ] 列表查询返回 `Flow<List<T>>`
- [ ] 至少实现一种有意义的查询功能（搜索、筛选、排序、联合查询或统计查询）
- [ ] 使用 `Room.databaseBuilder()` 或等价方式创建数据库实例
- [ ] 使用 DataStore 保存一个用户偏好或最近状态（如主题偏好、默认分类、最近搜索词等）

**参考 Lab：** Lab14（Room 基础）、Lab15（复杂查询）、DataStore 相关实验

---

### 3. 网络层

- [ ] 在 `AndroidManifest.xml` 中声明并实际使用 `android.permission.INTERNET`
- [ ] 使用 Retrofit、OkHttp、Ktor 或课程认可的其他方式发起网络请求
- [ ] 从真实 API 或 Mock API 获取数据，数据不得全部写死在客户端
- [ ] 至少有一个主要页面展示网络数据，或基于网络数据完成核心功能
- [ ] 网络请求通过 Repository 或独立数据源封装，UI / Composable 不直接发起请求
- [ ] 使用 Kotlin 协程处理网络请求
- [ ] 网络请求状态能在 UiState 中体现 Loading、Success、Error 中的至少几类
- [ ] 网络失败、空结果或数据格式异常时，有合理的错误提示或空状态
- [ ] 报告中说明 API 来源、接口地址、主要返回字段，以及这些数据在 App 中的用途

**参考 Lab：** 网络请求、协程、Repository 相关实验

---

### 4. 架构与状态管理

- [ ] 使用 ViewModel 管理 UI 状态
- [ ] 设计 UiState，能表达 Loading、Success、Error、Empty 中的至少几类状态
- [ ] 推荐使用 `sealed interface` / `sealed class` 或 data class 组织 UiState
- [ ] 使用 Repository 模式隔离数据来源
- [ ] ViewModel 通过 Repository 访问数据，不直接调用 DAO
- [ ] 使用 StateFlow 或 Flow 向 UI 暴露数据
- [ ] 使用 Kotlin 协程处理数据库、DataStore 和异步操作
- [ ] UI 通过 `collectAsState()` 或 `collectAsStateWithLifecycle()` 收集状态
- [ ] Composable 函数只负责界面展示和事件触发，不直接访问数据库或网络，不承担复杂业务逻辑

**参考 Lab：** Lab9（ViewModel）、Lab12（协程）、Lab13（Repository）

---

### 5. 功能完整性

应用必须具有完整的用户操作流程，不能只是静态展示。

至少需要从以下功能中实现 **2 项**：新增数据、编辑数据、删除数据、搜索数据、筛选数据、排序数据、收藏或取消收藏、查看详情、保存用户偏好、记录最近输入或最近访问内容。

同时需要满足：

- [ ] 应用可以在模拟器或真机上正常安装和运行
- [ ] 核心功能逻辑正确，无明显崩溃
- [ ] 有基本输入验证（必填项检查、数字范围检查或格式校验）
- [ ] 有合理错误提示（Snackbar、Toast、错误文本或空状态提示）
- [ ] 至少包含空状态、加载状态或错误状态中的一种
- [ ] 屏幕旋转后核心数据不丢失
- [ ] 点击系统返回键时行为符合预期

## 四、选做与提高项

选做项用于体现项目完成度和体验优化。基础网络请求已经属于必做项，不再作为选做加分。建议至少完成 **1–2 项**，功能必须真实可用，只有依赖项或空壳代码不计分。

### A 类：数据持久化增强

- [ ] 复杂数据库查询：联合查询、分组统计、模糊搜索（LIKE）、排序筛选等
- [ ] 数据导入 / 导出：支持导出 JSON 或 CSV，或从文件导入数据
- [ ] 数据库迁移：应用升级时支持新增表或字段，已有数据不丢失

### B 类：网络增强

- [ ] Coil 图片加载：使用 `AsyncImage` 加载网络图片
- [ ] 网络缓存：将网络数据保存到 Room，无网络时可以展示本地缓存
- [ ] 网络状态增强：检测网络连接状态，无网络时显示离线提示并允许重试
- [ ] 分页加载或下拉刷新：网络列表支持分页、刷新或重新加载
- [ ] WorkManager 后台任务：实现后台同步、提醒或定期任务

### C 类：高级 UI 与体验

- [ ] 大屏自适应布局：平板或横屏下实现列表-详情双栏等布局
- [ ] 动画效果：页面切换动画、列表项动画或自定义过渡效果
- [ ] 搜索体验优化：搜索防抖、自动补全或搜索历史
- [ ] 图表统计：使用图表库或 Compose 绘制统计图
- [ ] 下拉刷新或分页加载

### D 类：其他增强

- [ ] CameraX 拍照功能
- [ ] NotificationManager 本地通知或提醒
- [ ] 为 ViewModel 或 Repository 编写单元测试（至少 3 个用例）
- [ ] 使用 Hilt 或手动 AppContainer 实现依赖注入

## 十一、常见问题

**Q1：可以做游戏类应用吗？**

可以，但必须使用 Compose 构建界面，并满足数据存储和网络要求。例如游戏成绩、关卡进度、用户设置等应存入 Room 或 DataStore，同时可以接入在线题库、排行榜、每日提示或 Mock API 数据。

**Q2：可以几个人合作完成吗？**

不可以。期末大作业必须独立完成。如果发现雷同代码，将按课程要求处理。

**Q3：可以使用第三方库吗？**

可以，例如 Coil、Retrofit、图表库等。但必须在报告中说明用途。核心技术（Room、ViewModel、Navigation、DataStore 等）不能用第三方方案替代。

**Q4：必须做网络功能吗？**

必须。网络功能是本次大作业的必做项，至少需要从真实 API 或 Mock API 获取数据，并在应用核心页面或核心流程中使用。只添加 Retrofit、OkHttp、Ktor 等依赖但没有实际请求，不算完成网络要求。

**Q5：数据库设计需要多复杂？**

至少 2 张表，字段和关系要服务于应用场景。不要为了复杂而复杂，够用、清楚、能解释即可。

**Q6：功能简单会不会不及格？**

功能可以简单，但必须结构规范、数据真实持久化、网络请求真实可用、交互流程完整。功能复杂但 bug 多、无法运行或架构混乱，反而会扣分。

**Q7：UI 可以参考其他应用吗？**

可以参考设计风格和交互方式，但代码必须自己编写。直接复制他人代码视为抄袭。

**Q8：提交后还能修改吗？**

不能。提交后原则上不再接受修改、替换或补交，请在提交前确认 GitHub 仓库、`project_code.zip`、`report.md` 和截图都完整可用。
