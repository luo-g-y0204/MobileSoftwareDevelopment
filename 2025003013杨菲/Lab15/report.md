# Lab15：Flight Search 应用项目

## 1. Entity 实体设计说明
本次使用 Room Entity 映射两张数据表：
1. **Airport**：对应 `airport` 表，包含主键id、机场IATA代码、机场名称、年客流量，用于存储所有机场基础信息。
2. **Favorite**：对应 `favorite` 收藏表，自增主键id、出发地IATA、目的地IATA，仅存储航线代码。
额外定义 `Flight`、`FavoriteFlight` 数据类，用于接收**联表查询结果**，封装界面展示所需完整字段。

## 2. DAO 查询方法设计说明
DAO 接口 `FlightDao` 封装全部数据库操作：
1. **自动补全查询**：使用 `LIKE` 模糊匹配 IATA 代码和机场名称，按客流量降序排序，返回 Flow 实现实时观察。
2. **航班查询**：查询指定出发机场的所有目的地，通过条件排除出发地与目的地相同的航班。
3. **收藏查询**：使用 `INNER JOIN` 联合 `favorite` 与 `airport` 两张表，将代码转为机场名称展示。
4. **收藏增删**：提供插入、删除、按代码删除、判断是否收藏等方法，完成收藏状态切换。

## 3. SQL LIKE 关键字使用与作用
- **作用**：实现**模糊匹配搜索**，支持用户输入部分字符检索对应机场。
- **使用方式**：结合通配符 `%`，`%关键词%` 表示匹配**包含关键词**的字段。
- 实现逻辑：用户输入内容拼接为 `%xxx%` 传入 SQL，同时匹配 `iata_code` 和 `name` 两列。

## 4. 联合查询（JOIN）实现与作用
- **实现**：使用 `INNER JOIN` 将 `favorite` 表分别与出发机场、目的机场的 `airport` 表关联。
- **作用**：收藏表仅存储机场代码，通过联表查询将代码翻译成对应的机场名称，满足界面展示需求。

## 5. Preferences DataStore 使用场景与实现
- **使用场景**：持久化保存用户最后一次的搜索文本，实现应用重启状态保留。
- **实现流程**：
  1. 定义字符串 Preference Key；
  2. 封装 `saveSearchText` 挂起函数完成数据写入；
  3. 通过 Flow 读取本地存储数据，应用启动时自动回填搜索框。
- 特点：异步、线程安全，替代传统 SharedPreferences。

## 6. ViewModel 状态管理设计
ViewModel 作为业务与状态核心：
1. 使用 `MutableStateFlow` 管理搜索文本、选中机场等 UI 状态；
2. 结合 Flow 转换操作（debounce、flatMapConcat）实现**搜索防抖、自动查询**；
3. 统一封装数据库、DataStore 调用，所有耗时操作放在协程中执行；
4. 向外暴露只读 StateFlow，保证 UI 状态单向流转，配置变更（屏幕旋转）不丢失数据。

## 7. UI 界面切换逻辑说明
界面分为三种状态，根据**搜索文本+选中机场**自动切换：
1. 搜索框为空 → 展示收藏航线列表；
2. 搜索框有内容、未选择机场 → 展示自动补全建议列表；
3. 已选择机场 → 展示该机场出发的全部航班列表。
使用 Compose `LazyColumn` 实现列表滚动，封装通用条目组件复用布局。

## 8. 实验问题与解决过程
1. **问题1**：预置数据库加载失败
   解决：严格按照路径 `assets/database/flight_search.db` 放置文件，使用 Room `createFromAsset` 方法加载。
2. **问题2**：模糊查询无结果
   解决：忘记拼接 `%` 通配符，给查询参数前后添加 `%` 后正常匹配。
3. **问题3**：收藏状态不刷新
   解决：使用 Flow 监听数据库变化，数据变更自动通知 UI 刷新。
4. **问题4**：应用重启搜索文本丢失
   解决：正确初始化 DataStore，ViewModel 启动时读取历史文本并回填。
