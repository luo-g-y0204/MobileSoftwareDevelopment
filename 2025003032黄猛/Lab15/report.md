# Lab15 实验报告

## 1. Entity 设计说明

`Airport` 对应 `airport` 表，字段包含 `id`、`iata_code`、`name` 和 `passengers`。其中 `id` 作为主键用于唯一标识机场记录，`iata_code` 用于界面搜索和航线关联，`name` 用于展示机场名称，`passengers` 用于排序，方便优先展示热度更高的机场。

`Favorite` 对应 `favorite` 表，保存出发地和目的地的 IATA 代码，用于记录用户收藏的航线。为了避免重复收藏，同一组出发地和目的地通过联合唯一索引进行约束，这样用户多次点击收藏按钮时不会生成重复数据。

## 2. DAO 查询方法设计说明

`searchAirports()` 使用 `LIKE` 在 `iata_code` 和 `name` 中执行自动补全搜索，并按 `passengers` 降序排序。这个方法适合输入联想场景，用户只需要输入部分机场代码或名称就能获得候选项。

`observeRoutesFrom()` 查询指定出发机场到其他全部机场的航线。该查询使用交叉连接构造出发地和目的地的组合，再过滤掉同机场之间的无效记录，最后按照目的地热度排序，便于 UI 展示更有参考价值的航线。

`observeFavoriteRoutes()` 使用联合查询把收藏表和机场表关联起来，返回可直接显示的航线详情。它把收藏表中的代码转换成机场名称，避免界面层再去做二次查询，也让数据流更清晰。

## 3. `LIKE` 关键字的使用方法和作用

`LIKE` 用于模糊匹配字符串。本实验在查询时把用户输入包裹为 `"%关键字%"`，这样可以匹配机场代码或机场名称中包含该片段的记录。相比完全匹配，这种方式更符合搜索框的使用习惯，因为用户通常不会一次输入完整的机场名称。

在实际使用中，还需要注意大小写和特殊字符的处理。如果输入内容过短，可能会返回较多结果，因此界面通常会结合结果数量限制和排序规则来控制展示效果。

## 4. 联合查询的实现和作用

收藏表只保存机场代码，因此展示收藏列表时需要把 `favorite` 表和 `airport` 表关联起来，查询出出发地和目的地的机场名称。这样 UI 不必自己再做二次查询，也避免在 Compose 中写复杂的数据拼装逻辑。

联合查询的好处是数据转换集中在 DAO 层完成，界面拿到的就是完整的可展示对象。这样一方面减少代码重复，另一方面也降低了后续维护时修改数据结构的成本。

## 5. Preferences DataStore 的使用场景和实现

DataStore 用于保存搜索框文本。应用启动后从 `searchTextFlow` 读取数据并恢复输入内容，用户每次修改搜索框时即时写回，保证重启后状态可恢复。对于这种需要记住单个轻量状态的场景，DataStore 比传统 SharedPreferences 更适合，也更符合现代 Android 的推荐方案。

在实现上，读取和写入都通过协程完成，不会阻塞主线程。这样即使用户频繁修改输入内容，界面也不会明显卡顿。

## 6. ViewModel 状态管理设计

ViewModel 维护搜索文本、自动补全建议、当前航班列表和收藏列表，并通过 `StateFlow` 统一暴露给 Compose。界面只负责渲染状态和触发事件，这样数据流从数据库到 UI 的方向比较单一，便于排查问题。

这种设计的另一个好处是界面旋转或重建时不会丢失核心状态。即使 Activity 发生重建，只要 ViewModel 还在，搜索词、选中的机场和收藏信息都可以保持一致。

## 7. UI 切换逻辑说明

当搜索框为空时显示收藏列表；当用户输入并选择机场后显示该机场的航班列表；当输入但尚未选择时显示自动补全建议。这样可以把搜索、结果和收藏三种状态清晰分离。

从交互角度看，这种切换方式也减少了用户的操作成本。用户不需要额外切换页面，只要根据输入内容的变化，就能自然进入搜索、查看或收藏三种状态。

## 8. 实验中遇到的问题与解决过程

本实验的主要问题是预置数据库项目没有现成工程骨架，因此先搭建了可导入 Android Studio 的 Gradle 项目，再逐步补齐 Room、DataStore 和 Compose 层。随后通过统一的 ViewModel 状态管理把自动补全、航班展示和收藏功能串起来。

另一个容易出错的地方是数据库字段命名和 Kotlin 属性名的对应关系。为了保证查询结果能够正确映射到实体或投影类，需要在 SQL 中保持别名和数据类字段一致，这一点在联合查询时尤其重要。

在调试过程中，还需要反复检查收藏列表是否会出现重复项，以及搜索状态切换时界面是否会正确刷新。通过拆分 DAO、Repository 和 UI 三层职责，最终把这些问题逐个解决。

## 9. 实验结果总结

完成本实验后，应用能够支持机场搜索、航线列表展示、收藏航线记录以及输入状态持久化等功能。整体数据流清晰，界面响应也比较稳定，基本达到了实验要求。

从实现效果看，Room 负责本地数据库访问，DataStore 负责轻量状态保存，Compose 负责界面展示，三者配合后形成了较完整的移动端数据驱动架构。

## 10. 后续改进方向

如果后续继续扩展，可以增加收藏筛选、机场详情页和更完整的航线信息展示，比如起飞时间、票价区间和经停信息。

还可以进一步优化搜索体验，例如增加防抖输入、空状态提示和更细粒度的排序规则，让用户在较大数据集下也能快速找到目标机场。
